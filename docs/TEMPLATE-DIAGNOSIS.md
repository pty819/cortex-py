# 故障诊断事件样板（供下游 agent 对齐）

> 本文档是给**下游诊断 agent** 的输入样板。你的诊断过程（从观察→讨论→分析→定位→验证→回归）应该按这个结构组织后发给 cortex 入库。
>
> **为什么需要样板**：cortex 的抽取 prompt 按固定结构提取知识图谱。如果你的输入遵循本样板的结构和术语，抽取准确率和图谱连通性会显著提升——因为 prompt 里的实体类型、谓词、连接准则就是按这个样板设计的。
>
> **不是强制**：样板是"最佳实践"，不是死格式。实际诊断可能更简单或更复杂（循环、跳步），按实际情况走，但尽量覆盖样板里的关键要素。

---

## 一、完整故障事件时间轴样板

一个故障事件从发现到闭环，典型经历以下阶段。**实际中阶段可能循环或跳跃**（比如定位后又发现新线索回到讨论，或验证失败回到分析），这是正常的，如实记录每个阶段的动作和发现即可。

### 阶段 0：异常观察（Observation）
**什么时候**：操作员/监控系统/下游工艺首次发现异常的时刻。
**要记录**：
- 观察到的现象（征兆）：具体、可量化（"刻蚀速率从 1200A/min 降到 950A/min"，不是"速率不对"）
- 哪个传感器/工艺参数检测到的（`detected_by` / `monitored_by`）
- 观察时间、观察者（人/系统）
- 触发了什么告警/互锁（`triggers`）

**样板片段**：
```
[2026-06-15 02:14] EPD-1 端点检测信号斜率偏离基准 12%，触发"端点异常"告警。
                  观察者：自动监控系统 AMS。同时操作员张某注意到批次均匀性报表偏差。
```

### 阶段 1：问题界定与初步讨论（Scoping & Discussion）
**什么时候**：观察到异常后，团队界定问题范围、交换初步判断。
**要记录**：
- 界定的问题范围：影响哪个工艺步骤、哪个设备、哪个指标（`affects`）
- 初步生成的假设方向（`hypothesis`）：怀疑哪些子系统/哪些故障模式
- 讨论中涉及的已知约束（如"上周刚做过预防性维护，排除X"）
- 参与人员（`observed_by` / `person`）

**样板片段**：
```
[2026-06-15 02:30] 工艺工程师张某、设备工程师李某讨论：
                  - 影响：主刻蚀步骤的刻蚀速率和均匀性（affects 刻蚀速率、均匀性偏差）
                  - 假设1：怀疑气体输送系统（MFC 响应或管路微漏）
                  - 假设2：怀疑温控系统（基底温度漂移影响速率）
                  - 假设3：怀疑射频系统（功率稳定性影响等离子）
                  - 约束：射频系统上周校准合格，初步排除（ruled_out 射频系统——待数据确认）
```

### 阶段 2：数据采集与排查（Data Collection & Investigation）
**什么时候**：针对每个假设，采集相关传感器数据、做排查动作。
**要记录**（每个假设一组）：
- 排查动作（`diagnostic_action`）：查了什么数据、做了什么测试
- 检查的传感器/部件（`checked`）
- 发现（`found`）：数据模式、异常值、趋势
- 排除项（`normal` / `ruled_out`）：确认正常的项
- 信号特征模式（`signal_pattern`）：周期性、阶跃、漂移等

**样板片段**（假设1的排查）：
```
[2026-06-15 03:00] 排查气体输送系统：
                  - 动作：对比 MFC-1 设定值(50sccm)与实测值
                  - 发现：MFC-1 实测 46.2sccm，偏差 -7.6%（found MFC-1流量偏差）
                  - 信号模式：MFC-1 流量呈缓慢下降趋势，72h 内降 3.8sccm（signal_pattern）
                  - 检查 MFC-2/MFC-3：均在公差内（normal MFC-2、normal MFC-3）
[2026-06-15 03:15] 排查温控系统：
                  - 动作：调取 T-101(基底温度) 历史趋势
                  - 发现：T-101 从 80.0℃ 缓慢漂移到 82.3℃，72h 内 +2.3℃（found T-101漂移）
                  - 检查温控 PID 输出：正常跟踪（normal 温度PID）
[2026-06-15 03:30] 排查射频系统（验证排除）：
                  - 动作：检查 RF 功率、反射功率、匹配网络
                  - 发现：RF 正向 1500W、反射 <15W，均正常（normal 射频系统）
                  - 确认排除假设3（ruled_out 射频系统）
```

### 阶段 3：相关性分析与假设细化（Correlation & Refinement）
**什么时候**：多个排查有发现后，做交叉分析，找最强相关，细化假设。
**要记录**：
- 相关性分析（`correlates_with` / `no_correlation`）：哪些信号之间相关、相关系数
- 假设细化（`refines_to`）：从宽泛假设细化到具体方向
- 证据支持/反驳（`supports` / `contradicts`）

**样板片段**：
```
[2026-06-15 04:00] 相关性分析（取故障前 72h 数据）：
                  - MFC-1 流量偏差 与 刻蚀速率漂移：r=0.87（correlates_with，强相关）
                  - T-101 漂移 与 刻蚀速率漂移：r=0.41（correlates_with，弱相关）
                  - T-101 漂移 与 MFC-1 偏差：r=0.08（no_correlation，独立变量）
                  - 结论：MFC-1 流量偏差是主因，T-101 漂移是次要因素
                  - 假设细化：怀疑气体输送系统 --refines_to--> 怀疑MFC-1或管路微漏
                  - 证据支持：r=0.87 --supports--> MFC-1流量偏差是根因方向
```

### 阶段 4：根因定位（Root Cause Identification）
**什么时候**：细化假设后，做针对性验证，定位到具体根因。
**要记录**：
- 验证动作（`checked` / `found`）：校准记录、拆检、测试
- 排除项（`ruled_out` / `contradicts`）：验证过程中排除的假设
- 根因链（`caused_by` 多层）：从表面故障追溯到最底层根因
- 历史案例参照（`references` / `confirmed_by`）：相似历史案例

**样板片段**：
```
[2026-06-15 05:00] 验证 MFC-1 校准：
                  - 动作：用标准流量计校准 MFC-1
                  - 发现：校准在公差内（found MFC-1校准合格）
                  - 反驳：MFC-1校准合格 --contradicts--> 假设MFC-1校准漂移
                  - 排除：ruled_out MFC-1校准漂移
                  - 假设细化：怀疑MFC-1或管路微漏 --refines_to--> 怀疑管路连接处微漏
[2026-06-15 05:30] 验证管路微漏：
                  - 动作：对 MFC-1 下游管路做保压测试
                  - 发现：V-3 阀门处压降 0.3mTorr/min（found V-3阀门微漏）
                  - 历史参照：参考 2025-11 案例-007（相同故障模式：V-3 密封老化）
                  - 确认：V-3阀门微漏 --confirmed_by--> 案例-007
                  - 根因链：刻蚀速率漂移 --caused_by--> MFC-1流量偏差 --caused_by--> V-3阀门微漏 --caused_by--> V-3密封圈老化
```

### 阶段 5：修复措施（Remediation）
**什么时候**：定位根因后，执行修复。
**要记录**：
- 修复措施（`repaired_by`）：具体动作
- 修复后验证：故障是否消除、参数是否恢复

**样板片段**：
```
[2026-06-15 06:00] 修复：
                  - 措施：更换 V-3 阀门密封圈（repaired_by 更换V-3密封圈）
                  - 修复后验证：MFC-1 流量恢复 50.0sccm，T-101 趋势稳定
                  - 刻蚀速率恢复 1200A/min，均匀性偏差回到公差内
```

### 阶段 6：回归测试与预防（Regression & Prevention）
**什么时候**：修复后，跑回归批次确认稳定，并制定预防措施。
**要记录**：
- 回归测试结果：跑 N 批次，参数稳定情况
- 预防措施：维护计划调整、监控阈值优化、互锁逻辑更新
- 经验提炼：本次诊断的关键决策点、可复用的诊断模式

**样板片段**：
```
[2026-06-15 08:00] 回归测试：
                  - 跑 5 批次验证批次，刻蚀速率 1198-1205A/min，均匀性 ±1.8%（公差内）
                  - 结论：修复有效，设备恢复
[2026-06-15 09:00] 预防措施：
                  - 维护计划：V-3 密封圈更换周期从 12 个月缩短到 6 个月
                  - 监控优化：新增 MFC-1 流量偏差告警阈值（>3% 告警，>5% 互锁）
                  - 互锁更新：MFC 偏差>5% --triggers--> 工艺中断
                  - 经验提炼：本次诊断模式 = "速率漂移→查MFC偏差→查管路密封"，可复用
```

### 阶段循环说明
实际诊断中，**阶段 2-4 可能循环多次**：
- 阶段 4 验证失败 → 回到阶段 2 采集新数据 → 阶段 3 重新分析 → 阶段 4 新假设验证
- 阶段 5 修复后故障复发 → 回到阶段 1 重新界定问题

如实记录每次循环的完整路径（包括被排除的假设）——这些排除路径对未来的诊断极有价值。

## 二、机械结构-传感器-作用 样板

机械结构知识按以下层级组织，下游 agent 灌入结构文档时尽量覆盖这些层级和关系。

### 层级结构
```
equipment(设备/整机)
  └─ subsystem(子系统)
       └─ component(部件)
            ├─ sensor(传感器)        ← 监测这个部件
            └─ controller(控制器)    ← 控制这个部件
```

### 样板（以一个泛化的工艺模块为例）

```markdown
# 处理单元 PM-3

处理单元 PM-3 是用于 [工艺功能] 的精密设备，包含温控系统、真空系统、气体输送系统、射频系统四个子系统。

## 温控系统

温控系统负责维持 [工艺部位] 的温度稳定，由温度控制器、加热器和温度传感器组成。

### 加热器 H-1

加热器 H-1 安装在 [部位]，功率 2kW，由温度 PID 控制器调节功率输出。
- 监测：温度传感器 T-101（安装在 [部位]，量程 0-200℃，精度 ±0.1℃）
- 控制：腔体温度 PID（控制 H-1 功率，目标温度 80℃）
- 正常参数：T-101 读数 79.5-80.5℃，PID 输出 30-45%
- 常见故障：
  - 温控超调（T-101 超过 82℃）：PID 增益参数不当或加热器响应延迟
  - 温度振荡（T-101 振幅 >±1℃）：PID 整定不良或传感器噪声
  - 加热器失效（T-101 持续下降）：加热器断路或电源故障

## 真空系统

真空系统负责建立和维持工艺腔体的真空度，由干泵、分子泵和截止阀组成。

### 截止阀 V-3

截止阀 V-3 安装在 [管路位置]，控制 [气体/真空] 通断。
- 监测：压力传感器 P-02（安装在 V-3 下游，量程 0-1000Pa）
- 控制：PLC 主控（按工艺序列控制 V-3 开关）
- 正常参数：V-3 开启时 P-02 < 1mTorr，关闭时 P-02 上升率 < 0.1mTorr/min
- 常见故障：
  - 密封失效（P-02 上升率 >0.3mTorr/min）：密封圈老化或阀座损伤
  - 阀门卡涩（开关不到位）：机械磨损或异物卡阻
```

### 结构文档要提取的关系（agent 灌入时确保覆盖）
- `equipment --has_component--> subsystem`
- `subsystem --has_component--> component`
- `component --part_of--> subsystem`
- `sensor --installed_on--> component`（传感器装在哪）
- `component --monitored_by--> sensor`（部件被什么监测）
- `component --controlled_by--> controller`（部件被什么控制）
- `controller --regulates--> process_param`（控制器调节什么参数）
- `fault --affects--> component`（故障影响什么部件）
- `fault --has_symptom--> symptom`（故障什么征兆）
- `symptom --detected_by--> sensor`（征兆被什么检测）
- `process_step --depends_on--> process_step`（工艺步骤依赖）
- `process_step --configured_as--> process_param`（步骤配置什么参数）

## 三、问题排查定位思路样板

下游 agent 的排查过程按这个思路组织，能让 cortex 抽取出完整的迭代诊断推理链。

### 排查思路框架

```
1. 观察异常 → 界定问题（哪些指标异常、影响范围）
2. 生成假设（怀疑哪些子系统，基于经验/结构知识）
3. 逐个假设排查：
   a. 采集该子系统的传感器数据
   b. 分析数据模式（趋势/周期/阶跃/漂移）
   c. 判断该子系统是否异常
   d. 异常 → 进入相关性分析；正常 → 排除该假设（ruled_out）
4. 相关性分析：
   a. 对所有发现的异常信号做交叉相关
   b. 找最强相关的信号对
   c. 弱相关/无相关的标记为排除项（no_correlation）
5. 细化假设：
   a. 从宽泛假设（怀疑某子系统）细化到具体（怀疑某部件/某故障模式）
   b. 基于相关性证据支持（supports）或反驳（contradicts）细化方向
6. 验证根因：
   a. 针对细化假设做专项测试（校准/拆检/保压等）
   b. 查历史案例（references）看是否有相同模式
   c. 确认（confirmed_by）或排除（ruled_out）
7. 如验证失败 → 回到步骤 3 采集新数据（循环）
8. 定位根因 → 建立因果链（caused_by 多层）
9. 修复 → 验证修复有效
10. 回归测试 → 预防措施 → 经验提炼
```

### 排查记录样板（agent 发给 cortex 的文本格式）

下游 agent 把排查过程写成自然语言叙述（cortex 的抽取 prompt 会从中提取三元组）。**关键：每个排查动作、每个发现、每个排除项都要明确写出来**，不要只写结论。

**好的输入**（信息完整，抽取质量高）：
```
[03:00] 排查气体输送系统。对比 MFC-1 设定值 50sccm 与实测值，发现 MFC-1 实测 46.2sccm，
偏差 -7.6%。MFC-1 流量呈缓慢下降趋势，72h 内降 3.8sccm。检查 MFC-2 和 MFC-3 均在公差内，
正常。怀疑 MFC-1 校准漂移或管路微漏。

[04:00] 相关性分析：MFC-1 流量偏差与刻蚀速率漂移相关系数 0.87，强相关。T-101 漂移与
刻蚀速率漂移相关系数 0.41，弱相关。T-101 漂移与 MFC-1 偏差相关系数 0.08，无相关性，
是独立变量。结论：MFC-1 流量偏差是主因。假设从"怀疑气体输送系统"细化为"怀疑 MFC-1
或管路微漏"。

[05:00] 验证 MFC-1 校准。用标准流量计校准，MFC-1 校准在公差内，合格。这反驳了"MFC-1
校准漂移"的假设，排除该假设。假设细化为"怀疑管路连接处微漏"。对 MFC-1 下游管路做
保压测试，发现 V-3 阀门处压降 0.3mTorr/min，确认 V-3 阀门微漏。参考 2025-11 案例-007，
相同故障模式（V-3 密封老化），确认本次根因。根因链：刻蚀速率漂移由 MFC-1 流量偏差
引起，MFC-1 流量偏差由 V-3 阀门微漏引起，V-3 阀门微漏由 V-3 密封圈老化引起。
```

**差的输入**（只有结论，抽取质量低）：
```
排查发现是 V-3 阀门漏了，换了密封圈好了。
```
→ cortex 只能抽出 `V-3阀门微漏 --repaired_by--> 更换密封圈`，丢失了整个排查推理链、相关性分析、排除项、历史参照。

### 排查记录的关键要素清单
发给 cortex 前，agent 自查是否覆盖：
- [ ] 观察到的异常现象 + 检测传感器
- [ ] 生成的所有假设（包括后来被排除的）
- [ ] 每个假设的排查动作 + 发现 + 是否排除
- [ ] 相关性分析（相关系数 + 强弱判断）
- [ ] 假设细化路径（从宽泛到具体）
- [ ] 验证动作 + 确认/排除
- [ ] 历史案例参照
- [ ] 完整根因链（多层级）
- [ ] 修复措施 + 修复后验证
- [ ] 回归测试结果
- [ ] 预防措施

## 四、下游 agent 入库操作

整理好上述文本后，按以下方式入库（详见 `docs/GUIDE-INGEST.md`）：

### 故障排查时间轴 → 路径 B（自然语言叙述）
```bash
curl -X POST http://<cortex>:8002/v1/experience \
  -H "Content-Type: application/json" -H "X-Cortex-Actor: <agent>" \
  -d '{
    "scope": "mech:plant1/line:A/user:diag",
    "modality": "document",
    "content": {"kind": "text", "text": "<上面的排查记录文本>"},
    "context": {"observed_at": "2026-06-15T02:14:00Z", "intent": "incident_retrospective"},
    "idempotency_key": "case-2026-0615-rate-drift"
  }'
```
> `intent: "incident_retrospective"` 触发诊断因果抽取 prompt。

### 机械结构文档 → 路径 C（切块入库）
```bash
curl -X POST http://<cortex>:8002/v1/ingest/document \
  -H "Content-Type: application/json" -H "X-Cortex-Actor: <agent>" \
  -d '{
    "scope": "mech:plant1/line:A/user:diag",
    "text": "<上面的结构文档 markdown>",
    "intent": "structure"
  }'
```
> `intent: "structure"` 触发结构抽取 prompt。

### 已结构化的因果三元组 → 路径 A（直写，零损失）
